# Chinese Output Templates

Loaded when the review is rendered in Chinese. This file defines only the concrete rendering skeletons. Decision rules, evidence rules, coverage-led workflow, visual review rules, grading compatibility, and examples live elsewhere.

内部渲染规则（不要把本段文字输出给用户）：只有当所有 manifest/file-list 单元的文本内容已审查，且所有二进制、生成物、压缩产物、持久化输出、截断内容或不可读文件已直接检查或完成可复现 provenance 验证时，才允许写 `未审查变更：无`。若只是推测、约定一致或 presumed 已存在，必须列为建议验证、领域确认或审查限制。“重点发现”不是“阻断项专区”；非阻断但具有具体运行时、安全、数据、兼容性、运维或测试影响的问题仍应列入“重点发现”，并省略“阻塞原因”行。材料性候选问题如果不进入“重点发现”，必须在“提交建议”“建议验证”“建议补充说明”或“未审查变更”等正常栏目中可见处置；不要输出内部候选台账或本段规则说明。

## Default Developer Review

```markdown
# 提交前审查

**VERDICT:** <SAFE_TO_COMMIT | SAFE_TO_COMMIT_WITH_NOTES | DO_NOT_COMMIT>
**结论：** <简短、可执行的 1-2 句决策摘要：是否可提交 + 主要依据/最高风险 + 提交前必须动作或建议验证>
**统计：** <N 个阻塞项 · N 个非阻塞提醒 · N 个测试缺口 · N 个审查限制；四项均为 0 时写 无>
**差异来源：** <来源 + 精确范围：命令/PR/commit range/用户粘贴 diff；说明 staged/unstaged/基准分支；未知时写明未知>
**审查范围：** <完整审查 / 部分审查 + 已覆盖内容 + 未覆盖内容 + 是否影响 verdict>
**变更规模：** <已审查文本 diff：文件数> 个文件, +<新增行数> 行 / -<删除行数> 行；<如有生成/二进制/锁/迁移文件在此单独说明，无则省略>
**风险等级：** <🔴 高 | 🟡 中 | 🟢 低> - <风险依据：变更类型 + 路径重要性 + 高风险领域 + 测试充分性>
**未审查变更：** <无 | 逐项列出：对象 + 未审查原因 + 可能隐藏的风险 + 是否影响 verdict>

## 执行摘要

<按变更规模自适应长度：小改动 1-2 句话即可，多域大改动可用子列表展开各主线（如：**- 领域一：** 内容）。核心要求：说明变更意图、主要影响区域、是否发现阻断风险、提交前最重要的验证动作。若变更涉及多个独立领域，严禁将 (1)(2)(3) 堆在同一段落中，请用 Markdown 列表或加粗领域标签分条展示。不要重复下面"重点发现"和"变更概览"中已有的技术细节（注：此处对摘要的精简要求仅适用于执行摘要，不得以此为由简化"重点发现"和"变更概览"的深度与技术细节）。>

## 重点发现

1. [🔒|❌|⚠️|🧪|👁️|📈|🧭] <file:line-line | file:function | file:配置键 | file | 多文件契约> - <可操作问题标题>
   - 证据：<最短直接证据>
   - 影响：<受影响对象 + 触发条件 + 失败模式 + 最坏后果>
   - 修复：<代码级修改方向 + 边界条件 + 首选方案>
   - 验证：<最小验证闭环：具体命令/测试名/手动路径 + 覆盖场景 + 预期结果>
   - 置信度：<高 | 中 | 低> - <仅在非高置信度时说明原因>
   - 阻塞原因：<仅阻塞项包含此行>

只有在没有任何达到重点发现门槛的阻断项、非阻断风险、测试缺口或审查限制时，才写：

无。

## 提交建议

- **提交前必须处理：** <只列阻塞项；没有则写 `无`>
- **提交前建议处理：** <非阻塞但建议本次一起处理的问题；没有则写 `无`>
- **可后续跟进：** <不影响本次提交的维护性、文档、清理事项；没有则写 `无`>
- **建议验证：** <按优先级列出最小验证闭环：命令/测试名/手动路径 + 覆盖场景 + 预期结果；如无法运行，说明替代验证>
- **建议补充说明：** <PR 描述、变更原因、回滚说明、迁移说明、截图、监控说明；没有则写 `无`>

## 变更概览

- **修改：** <按责任域总结行为变化：业务逻辑/API/UI/配置/测试/文档/依赖>
- **新增：** <新增能力、入口、测试、配置、迁移、依赖及用途；没有则写 `无 - 仅修改。`>
- **删除：** <删除的行为、兼容层、配置、测试或资源；没有则写 `无。`>
- **行为变化：** <无用户/API/数据/权限/性能/发布行为变化 | 明确说明变化前、变化后、受影响对象和触发条件>

## 风险摘要

| 维度 | 结论 | 依据 |
|---|---|---|
| 正确性 | <通过 / 有风险 / 未充分验证> | <简短依据> |
| 安全与隐私 | <无明显风险 / 有风险 / 未充分验证> | <简短依据> |
| 数据与迁移 | <不涉及 / 有风险 / 未充分验证> | <简短依据> |
| 性能与扩展性 | <无明显风险 / 有风险 / 未充分验证> | <简短依据> |
| 兼容性 | <无破坏 / 有风险 / 未充分验证> | <简短依据> |
| 可观测性与回滚 | <充足 / 不足 / 不适用> | <简短依据> |
| 测试覆盖 | <充足 / 有缺口 / 未运行> | <简短依据> |

## 影响范围

- **直接影响：** <变更直接触达的模块、入口、用户路径、服务或任务>
- **间接影响：** <可能受影响的调用方、依赖、消费者、数据管道、部署流程；无则写 `无明显间接影响。`>
- **需要领域确认：** <无 | 领域 + 需要确认的问题 + 阻塞/非阻塞>

## 回归风险

**等级：** <🔴 高 | 🟡 中 | 🟢 低>
**原因：** <具体说明为什么是该等级>
**最小验证闭环：** <最少需要哪些测试或手动步骤才能降低风险>
```
